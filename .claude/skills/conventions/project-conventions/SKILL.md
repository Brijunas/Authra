# Project Conventions

This skill defines Authra's project structure, organization, and naming conventions.

## Solution Structure

```
Authra/
├── src/
│   ├── Authra.Domain/              # Entities, value objects, enums, exceptions
│   ├── Authra.Application/         # Service interfaces, DTOs, validators (feature folders)
│   ├── Authra.Infrastructure/      # EF Core, service implementations, persistence
│   └── Authra.Api/                 # Minimal API endpoints, middleware, Program.cs
├── tests/
│   ├── Authra.UnitTests/           # Domain logic, pure unit tests
│   └── Authra.IntegrationTests/    # Full API tests via WebApplicationFactory + Testcontainers
├── docker-compose.yml              # PostgreSQL 18 + Mailpit
├── CLAUDE.md                       # Architecture decisions, design context
├── CLAUDE-DATA-MODEL.md            # Entity specifications
├── CLAUDE-SESSION-TOKENS.md        # Token strategy
├── CLAUDE-API.md                   # API endpoint specifications
└── CLAUDE-RESEARCHED.md            # Research findings
```

## Feature Folder Organization

Application layer is organized by feature, not by technical concern:

```

│       └── ...

    │   ├── IUnitOfWork.cs
```

## File Naming

| Type | Pattern | Example |
|------|---------|---------|
| Entity | `{Name}.cs` | `User.cs`, `Tenant.cs` |
| Base entity | `Entity.cs`, `TenantEntity.cs` | Global/tenant base classes |
| Service interface | `I{Feature}Service.cs` | `IAuthService.cs` |
| Service impl | `{Feature}Service.cs` | `AuthService.cs` |
| DTOs | `{TypeName}.cs` (one type per file) | `LoginRequest.cs`, `LoginResponse.cs` |
| Validators | `{TypeName}Validator.cs` (one per file) | `LoginRequestValidator.cs` |
| Endpoints | `{Feature}Endpoints.cs` | `AuthEndpoints.cs` |
| EF Configuration | `{Entity}Configuration.cs` | `UserConfiguration.cs` |
| Migration | Auto-generated by EF Core | `{Timestamp}_{Name}.cs` |

## ID Prefix Convention

All IDs exposed in the API are type-prefixed. Raw UUIDs stored in the database.

| Prefix | Entity | Example |
|--------|--------|---------|
| `usr_` | User | `usr_019458a0-4e28-7def-8c12-fb3a4e5d6c7a` |
| `tnt_` | Tenant | `tnt_...` |
| `mbr_` | TenantMember | `mbr_...` |
| `org_` | Organization | `org_...` |
| `rol_` | Role | `rol_...` |
| `prm_` | Permission | `prm_...` |
| `inv_` | Invite | `inv_...` |
| `req_` | Request (tracing) | `req_...` |

Encoding/decoding handled by `IdPrefix` class at the API boundary.

## Permission Naming

Colon notation: `{action}:{resource}[.{subresource}]`

Examples: `accounts:read`, `roles:assign`, `organizations:members.write`

## API Versioning

All endpoints under `/v1` prefix:

```
/v1/auth/...
/v1/me/...
/v1/tenants/...
/v1/tenants/{id}/organizations/...
/v1/tenants/{id}/roles/...
/v1/permissions
```

## Migration Naming

EF Core migrations use descriptive layer names:

```
{Timestamp}_InitialIdentityLayer
{Timestamp}_TenantLayer
{Timestamp}_RbacLayer
{Timestamp}_SessionTokenLayer
```

## Environment Configuration

| Environment | Database | Email | Secrets |
|-------------|----------|-------|---------|
| Development | Docker PostgreSQL 18 | Mailpit (localhost:8025) | 1Password CLI (`op run`) |
| Test | Testcontainers PostgreSQL | InMemoryEmailSender | appsettings.Testing.json |
| Production | RDS PostgreSQL 18 | Resend SMTP | 1Password service account |

### Configuration Files

Base `appsettings.json` is committed to git as **boilerplate only** — it defines the config schema with empty/placeholder values. Never put actual secrets or environment-specific values in it.

Real configuration goes in `appsettings.Development.json` (gitignored) for local development and debugging.
